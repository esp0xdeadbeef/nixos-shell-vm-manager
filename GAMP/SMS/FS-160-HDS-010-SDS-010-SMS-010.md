# FS-160-HDS-010-SDS-010-SMS-010 Pin-refresh Construction Module

`modules/default.nix` emits `REFRESH_PINS`, `PIN_REFRESH_FLAKE_REF`,
`PIN_REFRESH_FLAKE_ATTRIBUTE`, and `PIN_REFRESH_LOCK_SCOPE` from evaluated,
assertion-checked instance options. The default lock scope is `host`, so every
pin-refresh-enabled VM backed by the consumer host flake uses one shared runtime
`flake.lock`. A VM backed by a custom flake repository shall declare a distinct
scope. `scripts/nixos-shell-vm-manager.sh` implements an eligible-event gate and
a best-effort construction subprocess.

The subprocess acquires the per-VM construction lock, the named shared
pin-refresh lock, and a shared construction token. It archives the approved
reference immutably, copies that source to a validated temporary workspace,
overlays the last successfully published lock from the selected scope, makes
only that copy writable, and updates it. It then archives and builds without
additional lock mutation,
computes a lock hash, and admits one validated output as `pin-refresh`. Only
after successful admission does it atomically publish the effective lock for
the next VM in the same scope. Cleanup closes file descriptors and removes the
workspace on both success and failure.

Failure Conditions:

- A disabled or excluded event invokes Nix.
- The immutable archived source or host lock is modified.
- Instances using the default host source refresh from independent per-VM lock
  state.
- A custom repository can read or publish the default host lock scope without
  an explicit matching scope declaration.
- A failed update, archive, build, validation, or admission publishes a shared
  lock.
- A failed update, archive, build, validation, or admission changes a slot.
- Failure prevents normal host-pinned selection.
- The output omits pin-refresh source or lock provenance.
- Construction bypasses the per-VM lock, named pin-refresh lock, or shared token
  pool.
- Rollback or recovery enters the module.
